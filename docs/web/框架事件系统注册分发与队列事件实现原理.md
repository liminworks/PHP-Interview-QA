# [L3] 框架事件系统如何注册监听器、分发事件？同步事件与队列事件的内部路径有何差异？

#### 一句话结论

事件系统维护事件类名到监听器列表的映射表；同步监听器直接调用 `handle()`，队列监听器将自身包装为 `CallQueuedListener` Job 推入队列延迟执行。

#### 体系讲解

**一、观察者注册流程**

Laravel 事件系统的核心数据结构：

```php
// Illuminate\Events\Dispatcher 内部（简化）
protected array $listeners = [
    OrderPlaced::class => [
        SendOrderConfirmation::class,
        UpdateInventory::class,
    ],
    UserRegistered::class => [
        SendWelcomeEmail::class,
    ],
];
```

**三种注册方式及其内部行为**：

| 方式 | 代码 | 适用场景 |
|---|---|---|
| `EventServiceProvider::$listen` 数组 | `protected $listen = [Event::class => [Listener::class]]` | 推荐，框架启动时批量注册 |
| 运行时注册 | `Event::listen(OrderPlaced::class, fn($e) => ...)` | 动态注册、测试场景 |
| EventSubscriber | `EventServiceProvider::$subscribe = [OrderSubscriber::class]` | 相关事件聚合在一个类 |

`$listen` 数组在 `EventServiceProvider::boot()` 中由框架遍历并调用 `Dispatcher::listen()`，最终写入 `$this->listeners`。

**EventSubscriber 与独立 Listener 的注册差异**：

- **独立 Listener**：一个类处理一种事件，在 `$listen` 数组中以值的形式出现；结构扁平，职责单一。
- **EventSubscriber**：一个类可订阅多个事件，框架通过调用 `Subscriber::subscribe(Dispatcher $events)` 方法，由订阅者自行调用 `$events->listen()` 批量注册：

```php
class OrderSubscriber
{
    public function subscribe(Dispatcher $events): void
    {
        $events->listen(OrderPlaced::class,   [self::class, 'onOrderPlaced']);
        $events->listen(OrderShipped::class,  [self::class, 'onOrderShipped']);
        $events->listen(OrderCancelled::class,[self::class, 'onOrderCancelled']);
    }
}
```

适合将同一业务聚合（如"订单相关事件"）的所有监听逻辑集中管理，避免 `$listen` 数组过于分散。

**二、事件分发流程**

`Event::dispatch(new OrderPlaced($order))` 内部执行路径：

```
1. 确定事件名：get_class($event) → 'App\Events\OrderPlaced'
2. 查找监听器：$this->listeners['App\Events\OrderPlaced']
3. 解析每个监听器（类名字符串 → 实例化，闭包直接使用）
4. 依次调用监听器的 handle($event) 方法
5. 若某个 handle() 返回 false → 停止传播（后续监听器不执行）
```

**三、同步 vs 队列事件的内部路径差异**

这是理解事件系统最关键的分叉点：

```
Event::dispatch(new OrderPlaced($order))
         │
         ▼
   解析监听器列表
         │
    ┌────┴─────────────────────────┐
    │ Listener 未实现 ShouldQueue  │ Listener 实现了 ShouldQueue
    │（同步路径）                  │（队列路径）
    ▼                              ▼
(new Listener)->handle($event)    $this->queueManager->push(
                                      new CallQueuedListener(
                                          Listener::class,
                                          'handle',
                                          [$event]
                                      )
                                  )
                                  // handle() 在 Worker 进程中异步执行
```

**队列路径的细节**：

- `CallQueuedListener` 是 Laravel 内部的 Job 包装器，它持有监听器类名、方法名和序列化的事件数据
- Worker 消费到该 Job 时，反序列化事件对象，实例化 Listener，调用 `handle()`
- 队列连接、队列名、重试次数、延迟时间均可在 Listener 上通过属性声明：

```php
class SendOrderConfirmation implements ShouldQueue
{
    public string $queue      = 'notifications';
    public int    $tries      = 3;
    public int    $delay      = 5;  // 延迟 5 秒
    public int    $backoff    = 10; // 重试退避秒数
}
```

#### 考察意图

考察候选人是否理解事件系统的"注册—分发—执行"三阶段分离，以及同步/异步的切换机制不是两套独立实现，而是在分发阶段根据 `ShouldQueue` 接口做路径分叉。同时考察 EventSubscriber 的注册原理，防止停留在"配置用法"层面。

#### 追问链

**Q1：EventSubscriber 和多个独立 Listener 在架构上如何选型？**

> 独立 Listener 遵循单一职责，便于测试和复用；但当一个业务领域对应多个事件（如订单的下单、发货、取消），相关逻辑分散在多个文件中，维护成本高。EventSubscriber 将同一聚合的监听逻辑集中管理，类似领域服务，适合事件数量 ≥ 3 且逻辑高度相关的场景。两者可混用，无需强制选一。

**Q2：队列监听器抛出异常时如何配置重试与失败处理？**

> 重试次数由 `$tries` 属性控制，超过后触发 `failed($event, $exception)` 方法（需在 Listener 中定义）；同时会向 `failed_jobs` 表写入失败记录。可通过 `$backoff` 属性配置指数退避重试间隔，或实现 `backoff()` 方法返回动态间隔数组（如 `[1, 5, 10]` 秒）。

**Q3：Laravel 的事件自动发现（`shouldDiscoverEvents()`）机制是如何工作的？与手动注册相比有什么代价？**

> 开启后，框架扫描 `app/Listeners` 目录，通过反射读取每个 Listener 的 `handle()` 方法参数类型，自动推断其监听的事件类并注册。优点是无需手动维护 `$listen` 数组；代价是每次请求都需要目录扫描和反射，生产环境建议运行 `event:cache` 将发现结果缓存为文件，消除运行时开销（类似路由缓存的机制）。

#### 易错点

1. **误以为 ShouldQueue 使"事件"异步**：`ShouldQueue` 是**监听器（Listener）**实现的接口，事件对象本身不受影响，仍然在当前请求同步创建和传递。只是"执行监听器 handle()"这一步被推迟到队列 Worker 中，事件对象需要可序列化。

2. **EventSubscriber 只写了 subscribe() 但忘记在 $subscribe 数组声明**：框架不会扫描所有类，只有在 `EventServiceProvider::$subscribe` 中声明的订阅者才会被调用 `subscribe()` 方法。忽略这一步会导致订阅者完全不生效且没有任何报错，排查困难。

#### 代码示例

```php
<?php
// 简化版 EventDispatcher，演示注册、分发、同步/队列路径分叉

interface ShouldQueue {}

class SimpleEventDispatcher
{
    /** @var array<string, list<string|callable>> */
    private array $listeners = [];

    public function listen(string $event, string|callable $listener): void
    {
        $this->listeners[$event][] = $listener;
    }

    public function dispatch(object $event): void
    {
        $eventName = get_class($event);
        $registered = $this->listeners[$eventName] ?? [];

        foreach ($registered as $listener) {
            // 实例化或直接使用闭包
            $instance = is_callable($listener) ? $listener : new $listener();

            if ($instance instanceof ShouldQueue) {
                // 队列路径：包装为 Job 推入队列（此处简化为输出提示）
                echo "[QUEUE] 推入队列: " . get_class($instance) . "::handle\n";
                continue;
            }

            // 同步路径：直接调用 handle()
            $result = is_callable($instance)
                ? $instance($event)
                : $instance->handle($event);

            if ($result === false) {
                echo "[STOP] 事件传播已中止\n";
                break;
            }
        }
    }
}

// --- 事件与监听器定义 ---

class OrderPlaced
{
    public function __construct(public readonly int $orderId) {}
}

class SendOrderConfirmation  // 同步监听器
{
    public function handle(OrderPlaced $event): void
    {
        echo "[SYNC] 发送确认邮件，订单 #{$event->orderId}\n";
    }
}

class UpdateInventory implements ShouldQueue  // 队列监听器
{
    public function handle(OrderPlaced $event): void
    {
        echo "[ASYNC] 更新库存，订单 #{$event->orderId}\n";
    }
}

// --- 注册与分发 ---
$dispatcher = new SimpleEventDispatcher();
$dispatcher->listen(OrderPlaced::class, SendOrderConfirmation::class);
$dispatcher->listen(OrderPlaced::class, UpdateInventory::class);

$dispatcher->dispatch(new OrderPlaced(orderId: 42));

/*
输出：
[SYNC] 发送确认邮件，订单 #42
[QUEUE] 推入队列: UpdateInventory::handle
*/
```
